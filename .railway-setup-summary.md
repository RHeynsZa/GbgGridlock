## 🚀 Deployment Flow

1. **Build Phase**
   - Railway builds the application using Nixpacks
   - Installs Python dependencies from `backend/pyproject.toml`

2. **Pre-Deploy Phase** (NEW)
   - Runs `python migrate.py` in a separate container
   - Executes all SQL migrations from `db/deploy/`
   - If migrations fail, deployment is aborted

3. **Deploy Phase**
   - Starts uvicorn FastAPI server
   - Connects to TimescaleDB
   - Optionally starts APScheduler worker (if `ENABLE_WORKER=true`)

4. **Health Monitoring**
   - Railway monitors `/health` endpoint every 100 seconds
   - Restarts service on failure (max 3 retries)

## 📋 Railway Environment Variables Required

### Mandatory
- `DATABASE_URL` - Automatically set by Railway TimescaleDB service

### Optional (Worker)
- `VT_CLIENT_ID` - Västtrafik API client ID
- `VT_CLIENT_SECRET` - Västtrafik API client secret
- `ENABLE_WORKER` - Set to `true` to enable polling worker

### Advanced (Optional)
- `WORKER_INTERVAL_SECONDS` - Polling frequency (default: 60)
- `WORKER_HTTP_CONCURRENCY` - Max concurrent requests (default: 5)
